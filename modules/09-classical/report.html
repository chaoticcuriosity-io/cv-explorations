09 · Classical computer vision

How it worked before deep learning — edges, contours, features, and flow

NoteNew here? Read this first

No prior knowledge needed — but this report is easier if you’ve skimmed Foundations (how a computer sees an image, what a model is, how we measure success). Stuck on a word? See the Glossary. Stuck on a model name? See Model families. Key terms are explained inline the first time they appear.

Why this module exists

Modules 01–08 are all neural networks. This one deliberately goes back to the pre-deep-learning toolkit — classical CV, meaning hand-designed algorithms (explicit math), with no training and no neural networks. These come from OpenCV (a free, widely used library of ready-made image-processing functions; see Model families) and are fast, transparent, deterministic, and need no training data or GPU. Deterministic means the same input always gives the same output — no randomness, no learned weights. They aren’t a historical curiosity — they still run inside camera calibration, visual SLAM front-ends (SLAM = building a map and tracking a camera’s position as it moves, using these classical features), image stabilisation, panorama stitching, and as cheap preprocessing for the deep models. Seeing them makes the rest of the curriculum legible: you can read every line of what these do, which is rarely true of a learned model.

Four classics

Edge detection & contours

Edge detection means finding the outlines where image brightness changes sharply. The Canny edge detector (a classic edge-finding recipe; see Model families) finds intensity discontinuities (gradients), then thins and links them into clean edges. A gradient is just how fast brightness changes from pixel to pixel — that change is what an edge detector looks for. Contours chain those edges into closed boundaries you can measure (area, perimeter, shape); a contour is a connected boundary line traced around a shape. For decades this was “object detection”.

Feature detection & matching (ORB)

ORB (a fast classical keypoint finder; see Model families) finds repeatable keypoints (corners) and encodes each as a binary descriptor, so the same physical point can be matched across two images. Here a feature or keypoint is a distinctive point — like a corner — that can be re-found in another image; a descriptor is the short numeric fingerprint of that keypoint used to match it across images; and feature matching is pairing the same physical point between two images. Matching keypoints between views is the engine behind panorama stitching, structure-from-motion, and the front-end of visual SLAM — the hand-built ancestor of the learned embeddings in module 07. (A related classical detector you may see named is SIFT, an older scale-invariant keypoint finder that ORB was designed to replace more cheaply.)

Optical flow

Optical flow is the per-pixel motion between two video frames — its direction and speed. Farneback dense optical flow (a classical algorithm that computes this flow for every pixel; see Model families) estimates a motion vector for every pixel between two frames — direction as hue, speed as brightness. It’s geometry, not recognition: no idea what is moving, only how. Flow feeds stabilisation, motion segmentation, and action recognition.

Results

Classical CV: operation timings (CPU)

Operation Latency (ms)
Canny edges 0.76
ORB detect+match 8.23
Farneback optical flow 96.94

All on CPU, no GPU, no training, no model download. Compare these millisecond timings to the deep models in modules 01–08 — classical CV is essentially free, which is why it still underpins calibration, SLAM front-ends, and stabilisation.

NoteWhat to notice
  • Essentially free. Canny runs in well under a millisecond; even dense optical flow is ~100 ms on a CPU with no model to load. Compare that to the VLM’s ~3.7 s per answer on a 24 GB GPU.
  • Transparent and deterministic. Every output follows from explicit math — no weights, no training set, same result every run. When you need to certify behaviour, that matters.
  • They still underpin modern systems. SLAM, calibration, and stitching lean on features and flow; deep models often sit on top of a classical geometric backbone, not instead of it.
  • The throughline. ORB→learned embeddings, Canny/contours→learned segmentation, optical flow→learned trackers: each deep module in this repo has a classical ancestor solving the same problem with hand-designed instead of learned features.

Where classical CV fails

  • Semantics — it finds edges and corners, never “cat” or “pedestrian”. No notion of meaning.
  • Robustness — sensitive to lighting, blur, texture, and parameter tuning (those Canny thresholds are hand-set).
  • High-level tasks — captioning, VQA, open-vocabulary anything are out of reach; that’s exactly the gap deep learning filled.
